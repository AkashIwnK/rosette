4.7 Solvers and Solutions

A solver is an automatic reasoning engine, used to answer queries about Rosette programs. The result of a solver invocation is a solution, containing either a binding of symbolic constants to concrete values, or an unsatisfiable core. Solvers and solutions may not be symbolic. Two solvers (resp. solutions) are eq?/equal? if they refer to the same object.

4.7.1 The Solver Interface and Classes

parameter

(current-solver) → (is-a?/c solver<%>)

(current-solver solver) → void?
  solver : (is-a?/c solver<%>)
The current-solver parameter holds the solver object used for answering solver-aided queries. If a query requires creation of additional temporary solvers, they all have the same class? as the current-solver. Supported solvers include kodkod% and, if installed, z3% and cvc4%.

Examples:

> (current-solver)

kodkod

> (require rosette/solver/smt/z3 rosette/solver/smt/cvc4 (only-in racket new))
> (current-solver (new z3%)) ; change the current solver
> (current-solver)

z3%

> (current-solver (new cvc4%)) ; change it again
> (current-solver)

cvc4%

interface

solver<%> : interface?

The solver interface specifies basic operations for posing and answering questions about the satisfiability of a set of formulas, expressed as (symbolic) boolean values. As a general rule, Rosette programs should not invoke these operations directly. The recommended way to access the solver is by posing solver-aided queries.

method

(send a-solver assert formula ...) → void?

  formula : boolean?
Adds the given formulas to the solver’s worklist.

method

(send a-solver clear) → void?

Clears the solver’s worklist.

method

(send a-solver solve) → solution?

Searches for a binding from symbolic constants to concrete values that satisfies all assertions in the solver’s worklist. If such a binding—or, a model—exists, it is returned in the form of a satisfiable (sat?) solution. Otherwise, an unsatisfiable (unsat?) solution is returned, but without computing an unsatisfiable core. A solution with a core can be obtained by calling debug on a-solver.

method

(send a-solver debug) → solution?

Searches for a minimal unsatisfiable core of the assertions in the solver’s worklist. If the worklist assertions are satisfiable, or a-solver does not support core extraction, an error is thrown. Otherwise, the result is an unsat? solution with a minimal core.

 (require rosette/solver/kodkod/kodkod) package: base

kodkod% : class?

  superclass: object%

  extends: solver<%>
A Rosette front-end to the Kodkod solver. This solver supports minimal core extraction.

 (require rosette/solver/smt/z3) package: base

A Rosette front-end to the Z3 solver from Microsoft. This solver does not support minimal core extraction.

 (require rosette/solver/smt/cvc4) package: base

class

cvc4% : class?

  superclass: object%

  extends: solver<%>
A Rosette front-end to the CVC4 solver from NYU. This solver does not support minimal core extraction.

4.7.2 Satisfiable and Unsatisfiable Solutions

A solution to a set of formulas consists of either a model, if the formulas are satisfiable, or a core, if they are not. The sat? and unsat? predicates recognize satisfiable and unsatisfiable solutions, respectively. A satisfiable solution can be used as a procedure: when applied to a bound symbolic constant, it returns a concrete value for that constant; when applied to any other value, it returns the value itself.

A solution supports the following operations:

procedure

(solution? value) → boolean?

  value : any/c
Returns true iff the given value is a solution.

procedure

(sat? solution) → boolean?

  solution : solution?
Returns true iff the given solution is satisfiable.

procedure

(unsat? solution) → boolean?

  solution : solution?
Returns true iff the given solution is unsatisfiable.

procedure

(sat binding) → solution?

  binding : (hash/c constant? any/c #:immutable #t)
Returns a satisfiable solution that holds the given binding from symbolic constants to values. The provided hashmap must bind every symbolic constant in its keyset to a concrete value of the same type.

procedure

(unsat) → solution?

(unsat a-core) → solution?
  a-core : (listof boolean?)
Returns an unsatisfiable solution. If a-core is provided, it must be a list of boolean values that are collectively unsatisfiable. Otherwise, the core of the produced solution is set to #f, to indicate that there is no satisfying solution but core extraction was not performed. (Core extraction is an expensive operation that is not supported by all solvers; those that do support it usually don’t compute a core unless explicitly asked for one.)

procedure

(empty-solution) → solution?

Returns a satisfiable solution with an empty binding as a model.

procedure

(model solution)

 → (or/c (hash/c constant? any/c #:immutable #t) #f)
  solution : solution?
Returns the binding stored in the given solution. If the solution is sat?, the binding is an immutable hashmap from symbolic constants to values. Otherwise, the binding is #f.

procedure

(core solution) → (or/c (listof (and/c constant? boolean?)) #f)

  solution : solution?
Returns unsatisfiable core stored in the given solution. If the solution is unsat? and a core was computed, the result is a list of boolean values that are collectively unsatisfiable. Otherwise, the result is #f.

procedure

(evaluate value solution) → any/c

  value : any/c
  solution : (and/c solution? sat?)
Given a Rosette value and a satisfiable solution, evaluate produces a new value obtained by replacing every symbolic constant c in value with (solution c) and simplifying the result.

Examples:

> (define-symbolic a b boolean?)
> (define-symbolic x y number?)
> (define sol
    (solve (begin (assert a)
                  (assert (= x 1))
                  (assert (= y 2)))))
> (sat? sol)

#t

> (evaluate (list 4 5 x) sol)

'(4 5 1)

> (define v (vector a))
> (evaluate v sol)

#(#t)

> (eq? v (evaluate v sol)) ; evaluation produces a new vector

#f

> (evaluate (+ x y) sol)

3

> (evaluate (and a b) sol)

b
